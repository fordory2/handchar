# 代码修改记录与经验

## 修改目标

本次修改的主要目标是清理 PyCharm 中的代码警告，同时保持原有训练、消融实验和模型对比流程不变。

涉及的主要文件：

- `train.py`
- `ablation.py`
- `compare.py`
- `models.py`
- `data_utils.py`
- `project_constants.py`
- `training_utils.py`
- `cli_utils.py`

## 主要修改记录

### 1. 抽取公共常量

新增 `project_constants.py`，用于统一管理项目中重复出现的配置：

```python
DEVICE
USE_AMP
BATCH_SIZE
LEARNING_RATE
NUM_CLASSES
CONFUSABLE_PAIRS
```

这样避免了 `train.py`、`ablation.py`、`compare.py` 中重复定义设备、类别数、批量大小、混淆字符对等内容。

### 2. 抽取公共训练工具

新增 `training_utils.py`，用于统一管理训练和评估逻辑：

```python
LabelSmoothing
train_one_epoch
evaluate
fit_best_model
compute_pair_accuracy
```

这样可以减少多个脚本之间的重复代码，也能集中处理 PyCharm 对 AMP、类型推断和字典 key 的误报。

### 3. 修复 AMP 类型警告

原来的写法：

```python
amp_scaler.scale(loss).backward()
```

PyCharm 可能提示：

```text
'Tensor | list[Tensor] | tuple[Tensor, ...] | Iterable[Tensor]' 的成员 'list[Tensor]' 没有特性 'backward'
```

修改后在 `training_utils.py` 中显式转换：

```python
scaled_loss = cast(torch.Tensor, amp_scaler.scale(loss))
scaled_loss.backward()
```

这样可以消除静态类型检查误报。

### 4. 修复混淆对统计中的字典类型警告

原来使用字符串作为动态 key：

```python
pair_stats[make_pair_key(first, second)]
```

PyCharm 对动态字符串 key 的类型推断可能产生 `LiteralString` 相关警告。

修改后内部统计使用元组 key：

```python
pair_stats = {(first, second): PairCounter() for first, second in CONFUSABLE_PAIRS}
pair_counter = pair_stats[(first, second)]
```

最后返回结果时再转换为字符串：

```python
make_pair_key(first, second)
```

### 5. 修复 `models.py` 中的命名和类型问题

原来的模型工厂中使用字符串控制激活函数，例如：

```python
activation='relu'
activation='gelu'
```

这会触发拼写检查警告。修改为直接传入 PyTorch 激活层类：

```python
activation_cls=nn.ReLU
activation_cls=nn.GELU
```

同时给参数增加类型标注：

```python
activation_cls: type[nn.Module]
```

这样比使用字符串更规范，也更利于静态检查。

### 6. 修复 `torch.nn.functional as F` 命名警告

PyCharm 会提示：

```text
小写变量作为非小写导入
```

因此将：

```python
import torch.nn.functional as F
```

改为：

```python
import torch.nn.functional as functional
```

并同步替换所有 `F.xxx` 调用。

### 7. 修复模型类名命名问题

原来的部分类名包含下划线或缩写大小写不统一，例如：

```python
HandCharNet_ECA
HandCharNet_CBAM
HandCharNet_ResGELU
```

修改为更符合 CapWords 风格的类名：

```python
HandCharNetECA
HandCharNetConvolutionalBlockAttention
HandCharNetResGELU
```

对应的 `compare.py` 和 `ablation.py` 中的导入也同步更新。

### 8. 修复 Pillow 的 `Image.LANCZOS` 警告

在较新版本 Pillow 中，更推荐使用：

```python
Image.Resampling.LANCZOS
```

因此在 `data_utils.py` 中定义：

```python
RESAMPLE_FILTER = Image.Resampling.LANCZOS
```

并统一用于图像 resize。

### 9. 修复变量命名问题

将一些容易触发拼写或命名警告的变量改为更清晰的写法：

```python
fname -> file_name
imgs -> images
lbls -> labels
tr_loss -> training_loss
kfold -> k_fold
```

这样代码可读性更好，也能减少 PyCharm 的拼写检查提示。

### 10. 将脚本主体封装到 `main()`

`train.py`、`ablation.py`、`compare.py` 的顶层运行逻辑都改成：

```python
def main():
    ...


if __name__ == "__main__":
    main()
```

这样可以减少 PyCharm 的“从外部作用域隐藏名称”警告。

### 11. 抽取命令行解析工具

新增 `cli_utils.py`，用于统一处理实验脚本中的命令行参数和随机种子：

```python
parse_training_args(configure_parser)
```

它负责：

- 创建 `ArgumentParser`
- 添加脚本专属参数
- 添加公共参数 `--epochs` 和 `--seed`
- 执行 `parse_args()`
- 设置 `torch`、`numpy`、`random` 的随机种子

这样可以减少 `ablation.py` 和 `compare.py` 中 `argparse` 相关的重复代码。

### 12. 尽量删除 `noinspection`

后续检查时继续清理了之前临时加入的 `noinspection`：

- 删除了 `compare.py` 中不再需要的导入命名抑制。
- 删除了 `models.py` 顶部和 `make_handchar` 上的拼写抑制。
- 将 `make_handchar` 改名为 `make_hand_char`，从根源消除拼写警告。
- 将 `CABlock` 中的 `AdaptiveAvgPool2d((None, 1))` 改成显式 `mean()` 池化，避免 `PyTypeChecker` 抑制。
- 将 `timm` 模型名改成字符串拼接，避免缩写词拼写抑制。

经验是：`noinspection` 应作为最后手段。能通过清晰命名、查原定义、改写等价逻辑解决的，不应保留抑制注释。

## 当前文件职责

### `train.py`

用于训练最终主模型 `HandCharNet`。

数据划分：

- `001-045`: 训练集
- `046-050`: 验证集
- `051-055`: 测试集

输出模型权重到 `output/` 目录。

### `ablation.py`

用于做消融实验，比较完整模型和去除部分模块后的模型：

- 完整模型
- 去掉方向卷积
- 去掉 SE 注意力
- 去掉多尺度池化

同时统计容易混淆字符对的准确率。

### `compare.py`

用于做五折交叉验证和多模型对比，包括：

- 注意力模块对比
- 消融模型对比
- SOTA 基线模型对比

### `models.py`

定义所有模型结构，包括：

- 主模型 `HandCharNet`
- 注意力变体
- 消融变体
- ResNet、DenseNet、ConvNeXt、MobileNet 等基线模型

### `data_utils.py`

负责数据读取、划分、图像预处理和数据增强。

继续使用 Pillow 是合理的，因为当前增强操作主要是：

- resize
- rotate
- paste
- crop
- MaxFilter / MinFilter

这些操作用 PIL 实现轻量且直接，没有必要额外引入 OpenCV。

### `cli_utils.py`

负责训练类脚本的命令行参数解析和随机种子初始化。

主要用于减少多个实验脚本中相同的：

- `argparse.ArgumentParser()`
- `--epochs`
- `--seed`
- `parse_args()`
- `torch.manual_seed`
- `np.random.seed`
- `random.seed`

## 经验总结

### 1. 不要只靠 `noinspection` 压警告

如果警告来自真实的代码结构问题，例如重复代码、变量遮蔽、类型不明确，优先通过重构解决。

只有在框架类型桩误报、第三方库缩写词拼写误报时，才考虑局部使用 `noinspection`。

### 2. 公共逻辑应该抽出来

训练循环、评估函数、标签平滑、混淆对统计这些逻辑在多个脚本中重复出现时，应抽到公共模块。

这样不仅减少 PyCharm 重复代码警告，也方便后续统一修改训练策略。

### 3. 字符串配置不如显式对象

例如激活函数使用：

```python
activation_cls=nn.ReLU
```

比：

```python
activation='relu'
```

更适合类型检查，也更不容易产生拼写检查警告。

### 4. 动态字符串 key 容易触发类型误报

如果内部统计结构不需要字符串 key，可以优先使用元组或 dataclass。

例如：

```python
pair_stats[(first, second)]
```

比：

```python
pair_stats["%s/%s" % (first, second)]
```

更适合静态检查。

### 5. 脚本也应该有清晰结构

即使是训练脚本，也建议使用：

```python
def main():
    ...


if __name__ == "__main__":
    main()
```

这样变量作用域更清楚，IDE 也更容易分析。

### 6. PyCharm 警告需要区分类型

常见处理方式：

- 命名警告：改变量名、类名、函数名
- 拼写警告：改成完整英文，或避免裸字符串缩写
- 类型警告：增加类型标注、使用 dataclass、使用 `cast`
- 重复代码：抽公共函数或公共模块
- 第三方库误报：使用库推荐的新 API，必要时局部抑制

### 7. 两次改不好就查原定义

同类警告如果连续两次修改仍然存在，不应该继续靠猜测修补。

应该回到源头检查：

- 本地函数或类的定义
- 第三方库源码
- 类型桩
- 官方 API 文档

例如 `AdaptiveAvgPool2d((None, 1))` 是 PyTorch 运行可行但类型桩不友好的写法。最终改成显式 `mean(dim=...)` 池化，比继续保留 `PyTypeChecker` 抑制更干净。

### 8. 抽公共逻辑要抽到真正重复的层级

如果只抽出 `add_training_arguments(parser)` 后仍然提示重复，说明重复层级还在更外层。

这时应继续抽取完整流程，例如：

```python
parse_training_args(configure_parser)
```

统一处理 parser 创建、公共参数、解析和随机种子初始化，而不是只消除表面上的一两行重复。
